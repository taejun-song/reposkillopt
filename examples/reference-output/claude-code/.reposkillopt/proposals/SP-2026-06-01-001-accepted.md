## Proposed text

Add the following sentence to **Repository Understanding Workflow** stage **(c) "Inspect manifests, configs, tests, deployment files, and entrypoints"**, after the existing list of files to read:

> Enumerate **every** subdirectory of the test root (e.g., `tests/typing/`, `tests/integration/`, `tests/property/`), not only the top-level `test_*` files — runner-specific trees (typing, doctest, property, integration) are easy to miss and are often the ones that break on change.

And add the following sentence to stage **(d) "Map modules, layers, dependencies, domain concepts, and data flow"**, after the existing instruction:

> Assign a role to **internal/underscore-prefixed modules** as well as public ones; secondary structures carry the implementation detail a maintenance reader most needs.

## Justification

Two `candidate-for-generic` Feedback Items recur on the same pattern: `FB-2026-05-31-002` (the workflow inspected `tests/` and stopped, missing `tests/typing/`) and `FB-2026-06-01-004` (the spec was rated strongest precisely *because* it enumerated internal `_`-prefixed modules — a quality that should be reliable, not lucky). Both reduce to one generalizable rule: **do not stop at the obvious public/top-level surface; enumerate secondary structures.** That belongs in the canonical workflow, not in a single Repository Specification, because the failure mode reappears on every repository with internal modules or non-pytest test trees. The edit is additive and reviewable in well under five minutes.
