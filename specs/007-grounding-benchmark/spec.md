# Feature Specification: Grounding Benchmark Harness

**Feature Branch**: `007-grounding-benchmark`
**Created**: 2026-06-07
**Status**: Draft
**Input**: User description: "A reproducible benchmark that measures how grounded the skill's Repository Specifications are — the fraction of cited claims that resolve against the real repo files — across a pinned suite of repositories, emitting a report under rubric/benchmarks/."

## Clarifications

### Session 2026-06-07

Resolved as informed defaults (delegated; recorded for explicitness):

- **Headline metric** = citation-resolution rate (resolved citations / resolvable citations), per the feature-005 grounding rule (`cmd:`/`output:` excluded; malformed counts as unresolved). Reported alongside the 7 deterministic checks and coverage counts (sections present, labeled claims, citations).
- **Reproducibility** = repos are pinned by `url@commit` (or a local path); a run over the same manifest + same specs yields identical numbers because the metric is computed without a model.
- **Scoring mode (default)** scores a **pre-written** spec against its repo (no LLM). **Generate mode** (opt-in) first regenerates the spec via the engine for the current canonical skill, then grounds it — LLM-dependent, outside the deterministic acceptance, validated by running.
- **Seed suite** uses repositories the project already ships reference specs for (e.g. `pallets/click` and the held-out reference repos), pinned by commit.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Measure grounding across a repo suite (Priority: P1)

A maintainer runs the benchmark over a pinned suite of repositories and gets a per-repo and aggregate **citation-resolution rate** plus deterministic-check results — a defensible number for validation and promotion ("X% of cited claims resolve").

**Why this priority**: This is the core deliverable — a reproducible, model-free measurement of the skill's central value claim (evidence grounding). Everything else supports or extends it.

**Independent Test**: Run the benchmark against a small manifest of (repo, spec) pairs with known citations; confirm the reported per-repo resolution rates match hand-computed values and the aggregate matches.

**Acceptance Scenarios**:

1. **Given** a manifest of (repo, spec) entries, **When** the benchmark runs, **Then** for each entry it reports resolved/total citations, the resolution rate, and each of the 7 deterministic checks.
2. **Given** the per-entry results, **When** the run finishes, **Then** an aggregate is reported (mean and median resolution rate; share of entries passing all deterministic checks).
3. **Given** the same manifest and specs are run twice, **When** results are compared, **Then** every number is identical (model-free, reproducible — SC-002/SC-005).

### User Story 2 - Reproducible, shareable report (Priority: P1)

The run writes a dated report under `rubric/benchmarks/` containing the per-repo table, the aggregate, and the exact repo commit pins used — so the result can be cited, re-run, and trusted.

**Why this priority**: A benchmark is only useful if its output is durable, reproducible, and shareable; the report is what gets referenced in docs and promotion.

**Independent Test**: After a run, confirm a report file exists with the per-repo rows, the aggregate, and the commit pins, and that a machine-readable summary is included.

**Acceptance Scenarios**:

1. **Given** a completed run, **When** the report is opened, **Then** it lists each repo with its commit pin, resolution rate, deterministic-check outcome, and the aggregate.
2. **Given** the report, **When** a reader wants to reproduce it, **Then** the manifest pins (and the command) needed to reproduce are recorded in the report.

### User Story 3 - Benchmark the current skill, not just shipped specs (Priority: P2)

A maintainer runs the benchmark in **generate mode**: the spec for each repo is regenerated by the current canonical skill (via the engine) and then grounded — measuring the skill's present grounding rather than the pre-written reference outputs.

**Why this priority**: Closes the loop between the skill and the benchmark, so the number reflects the live skill. Depends on Story 1's scoring/report and is LLM-dependent (outside deterministic acceptance).

**Independent Test**: Run generate mode against one small repo; confirm a fresh spec is produced via the engine and grounded, and the same per-entry metrics are reported.

**Acceptance Scenarios**:

1. **Given** generate mode and a repo, **When** the benchmark runs, **Then** a spec is regenerated via the engine and grounded, and its resolution rate is reported.
2. **Given** generate mode is unavailable (no model/provider), **When** the user runs default mode, **Then** the benchmark still scores pre-written specs with no model.

### Edge Cases

- **Repo unavailable** (clone fails / path missing): the entry is recorded as an error in the report (not a crash); the run continues and the aggregate notes how many entries were skipped.
- **Spec with zero citations**: reported as a defined rate (1.0 over zero resolvable, or explicitly "no citations"), not a divide-by-zero.
- **Empty / malformed manifest**: a clear error; nothing is written.
- **Commit pin no longer fetchable**: recorded as an error for that entry; others proceed.
- **Large repo**: grounding reads only the files the spec cites; runtime scales with citation count, not repo size.
- **Offline (default mode, local repos)**: scoring pre-written specs against already-present repos requires no network; only `url@commit` entries need network to clone.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The benchmark MUST accept a **manifest** listing entries, each identifying a repository (local path or `url@commit`) and a Repository Specification to score.
- **FR-002**: For each entry, the benchmark MUST compute the **citation-resolution rate** and the **7 deterministic checks** by resolving the spec's citations against the actual repository files, with **no language model** involved (default mode).
- **FR-003**: For `url@commit` entries, the benchmark MUST make the repository available at that exact commit (e.g. a pinned shallow clone) before scoring; local-path entries are used as-is.
- **FR-004**: The benchmark MUST produce an **aggregate**: mean and median resolution rate across entries, and the share of entries passing all deterministic checks.
- **FR-005**: The benchmark MUST write a **report** under `rubric/benchmarks/` containing the per-entry results, the aggregate, and the repository commit pins used (for reproducibility), plus a machine-readable summary.
- **FR-006**: Re-running the same manifest against the same specs MUST yield **identical numbers** (the metric is deterministic and model-free).
- **FR-007**: The benchmark MUST ship an initial **seed manifest** of repositories the project already has reference specs for, pinned by commit, scoring the shipped reference specs.
- **FR-008**: The benchmark MUST offer an opt-in **generate mode** that regenerates each entry's spec via the engine for the current canonical skill, then grounds it (LLM-dependent; not part of the deterministic acceptance).
- **FR-009**: A failed or unavailable entry MUST be recorded as an error and MUST NOT abort the whole run; the aggregate MUST note skipped entries.
- **FR-010**: The benchmark MUST reuse the existing grounding, evidence-pack, spec-generation, and rubric components and MUST NOT add a heavyweight runtime dependency (a pinned clone uses standard tooling).
- **FR-011**: The benchmark MUST NOT modify the canonical skill, the rubric definitions, or feature-005 behavior.
- **FR-012**: The benchmark MUST be runnable from the engine's command surface with a manifest argument and a mode selector (score pre-written vs generate).

### Key Entities *(include if feature involves data)*

- **Benchmark manifest**: the pinned list of entries `{name, repo (path or url@commit), spec path}` that defines a reproducible run.
- **Benchmark entry result**: per-repo outcome — resolved/total citations, resolution rate, the 7 deterministic checks, concrete failures, and any error.
- **Benchmark report**: the dated artifact under `rubric/benchmarks/` — per-entry table + aggregate + commit pins + machine-readable summary.
- **Aggregate**: mean/median resolution rate and the pass-share across entries.
- **Mode**: score-pre-written (default, model-free) vs generate (opt-in, model-backed).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For a manifest of (repo, spec) pairs with known citations, the reported per-repo resolution rates equal the hand-computed values, and the aggregate equals the mean/median of those.
- **SC-002**: Two runs of the same manifest + specs produce byte-identical metric values (reproducible).
- **SC-003**: The benchmark scores pre-written specs with **zero** model calls (the grounding metric is reproducible from the repo and spec text alone).
- **SC-004**: A run over the seed suite emits a `rubric/benchmarks/` report containing every entry's commit pin, its resolution rate and deterministic-check outcome, and the aggregate.
- **SC-005**: An entry whose repo cannot be made available is reported as an error and does not abort the run; the aggregate states how many entries were skipped.
- **SC-006**: The benchmark adds no heavyweight runtime dependency and leaves the canonical skill, rubric, and feature-005 behavior unchanged.

## Assumptions

- The feature-005 grounding (`citation resolution`, the 7 deterministic checks) is the scoring engine and is reused unchanged; its resolution rule (forms, `cmd:`/`output:` exclusion, malformed→unresolved) is the benchmark's definition of "resolved."
- Pinning repositories by commit is sufficient for reproducibility of the deterministic metric; the spec text being scored is also fixed (a file in the repo for default mode).
- The seed suite leans on repositories the project already references (reference-output specs exist for them); availability is via pinned clone or local checkout.
- Generate mode reuses the engine's provider layer (keyless `claude-cli` / API-key / local), consistent with feature 005, and is validated by running, not by deterministic unit tests.
- Reports accumulate under `rubric/benchmarks/` (dated files); they are working artifacts, not part of the canonical skill.
